RAG System
Implementation Challenge

RAG 系统实现挑战 — Full-Site Knowledge Base + AI Chatbot

Build a production-grade Retrieval-Augmented Generation (RAG) chatbot that can answer questions about an entire website corpus. This challenge tests your ability to architect, implement, and deploy a complete AI-powered knowledge retrieval system.

构建一个生产级的检索增强生成（RAG）聊天机器人，能够回答关于完整站点语料库的问题。 本挑战测试你设计、实现和部署完整 AI 驱动知识检索系统的能力。

📋 Challenge Overview / 挑战概述

You are required to build a RAG-based question-answering system that ingests, indexes, and retrieves information from a real Chinese university website. The system must handle bilingual content (Chinese + English), support conversational queries, and provide accurate, citation-backed answers.

你需要构建一个基于 RAG 的问答系统，从真实的中国大学网站中摄取、索引和检索信息。 系统必须处理双语内容（中文+英文），支持对话式查询，并提供准确的、有引用支持的答案。

Source Website / 源站点
www.thss.tsinghua.edu.cn
Site Name / 站点名称
清华大学软件学院
Tsinghua School of Software
Corpus Size / 语料规模
~850 pages / ~850 页
Content Types / 内容类型
News, announcements, faculty info
新闻、公告、师资信息
Media Assets / 媒体资产
Text + Images / 文本+图片

Full-site crawl
全站抓取（所有栏目）
Corpus Structure / 语料结构：
The site is organized into 5 main columns with paginated list pages. Each article contains: title, date, summary, body text, and accompanying images. Key columns: 新闻动态 (~650 articles), 学生动态 (~70), 学院风采 (~40), 科研成果, 招聘信息.

站点分为 5 个主要栏目，带分页列表。每篇文章包含：标题、日期、摘要、正文、配图。 主要栏目：新闻动态（~650篇）、学生动态（~70）、学院风采（~40）、科研成果、招聘信息。

✅ Required Deliverables / 必需交付物

1. Deployed RAG Chatbot with Authentication / 已部署的 RAG 聊天机器人（带认证）

You must deploy a working web application with the following requirements:

你必须部署一个具有以下要求的工作 Web 应用：

  • Login authentication: Username/password login gate before access to chatbot (simple is fine, even hardcoded credentials for demo)
    登录认证：聊天机器人访问前的用户名/密码登录门槛（简单即可，demo 用硬编码凭证也行）
  • Chat interface: User-friendly conversational UI with message history
    聊天界面：用户友好的对话 UI，带消息历史记录
  • RAG pipeline: Must actually retrieve from indexed corpus and generate answers (not just an API wrapper)
    RAG 管道：必须真正从索引语料库检索并生成答案（不仅仅是 API 包装器）
  • Citation/source links: Each answer should cite which article(s) it drew from
    引用/来源链接：每个答案应引用其来源文章
  • Publicly accessible URL: Your chatbot must be accessible via HTTPS on the provided server (123.59.90.15:8443). IP address access is sufficient (no domain name required).
    公开可访问 URL：你的聊天机器人必须通过 HTTPS 在提供的服务器（123.59.90.15:8443）上访问。IP 地址访问即可（无需域名）。
⚠️ Critical Requirement / 关键要求：
The chatbot MUST be behind authentication. You must provide test credentials (username + password) in your submission email so evaluators can access your system. Systems without login will not be considered complete.

聊天机器人必须有认证保护。你必须在提交邮件中提供测试凭证（用户名+密码），以便评估者访问你的系统。没有登录的系统将被视为不完整。

2. Public GitHub Repository / 公开 GitHub 代码仓库

You must provide a public GitHub repo containing:

你必须提供一个公开 GitHub 仓库，包含：

  • Complete source code: All crawlers, indexing scripts, backend API, frontend UI
    完整源代码：所有爬虫、索引脚本、后端 API、前端 UI
  • README.md: Clear setup instructions, architecture diagram, technology stack
    README.md：清晰的设置说明、架构图、技术栈
  • requirements.txt / package.json: All dependencies listed
    requirements.txt / package.json：列出所有依赖项
  • Configuration examples: .env.example or config templates (no secrets committed)
    配置示例：.env.example 或配置模板（不提交密钥）
  • License: Open-source license (MIT, Apache 2.0, etc.)
    许可证：开源许可证（MIT、Apache 2.0 等）
Why public repo? / 为什么要公开仓库？
TKE evaluators will review your code for quality, architecture decisions, and adherence to best practices. This is as important as the working demo. Private repos will not be accepted unless explicitly arranged in advance.

TKE 评估者将审查你的代码质量、架构决策和最佳实践的遵循情况。这与工作演示同等重要。 除非提前明确安排，否则不接受私有仓库。

📊 Evaluation Criteria / 评估标准

  1. Correctness (40%) / 正确性： How accurately does the system answer questions from the evaluation question set?
    系统回答评估问题集的准确度如何？
  2. Code Quality (30%) / 代码质量： Clean architecture, proper error handling, documentation, testability
    清晰的架构、适当的错误处理、文档、可测试性
  3. System Design (20%) / 系统设计： Scalability, modularity, choice of tech stack, chunking/embedding strategy
    可扩展性、模块化、技术栈选择、分块/嵌入策略
  4. UX & Deployment (10%) / UX 与部署： Usability of the chat interface, deployment robustness, auth implementation
    聊天界面可用性、部署稳定性、认证实现

🚫 Rules & Constraints / 规则与约束

  • Ethical crawling: Respect robots.txt, use polite rate limiting (1-2 req/sec), set proper User-Agent
    合规爬取：遵守 robots.txt、使用礼貌限速（1-2 req/sec）、设置合适的 User-Agent
  • No data redistribution: You may index the content for your RAG system, but do not publicly republish the raw corpus
    不得数据再分发：你可以索引内容用于 RAG 系统，但不得公开重新发布原始语料库
  • Original work: You must write your own code. You may use libraries/frameworks, but not copy entire RAG implementations from tutorials
    原创作品：你必须编写自己的代码。可以使用库/框架，但不得从教程复制完整 RAG 实现
  • LLM usage: You may use any LLM API (OpenAI, Anthropic, open-source models) for generation, but the RAG pipeline must be your implementation
    LLM 使用：你可以使用任何 LLM API（OpenAI、Anthropic、开源模型）生成，但 RAG 管道必须是你的实现
